VULN-001: Unvalidated File Upload
------------------------------------------------------------------------------
Severity: CRITICAL
Status: CONFIRMED
Location: app.py:283-303 (ics_import function)

Description:
The /ics/import endpoint accepts file uploads without proper validation. While
the current implementation only simulates import (no file is saved), the code
structure is vulnerable if file saving is implemented.

Evidence:
    file = request.files['datafile']
    if file.filename == '':
        flash('No file selected', 'danger')
        return redirect(url_for('ics_import_page'))
    # No extension validation
    # No MIME type validation
    # No file size limit
    # No content sanitization

Risk:
- Remote Code Execution if files are saved to web-accessible directory
- Denial of Service via large file uploads
- Malicious file storage even if not executable

Mitigation Strategy:
1. Validate file extension against whitelist (.json, .csv only)
2. Validate MIME type matches expected content
3. Enforce maximum file size limit (e.g., 10MB)
4. Validate file content structure before processing
5. Store uploaded files outside web root with randomized names
6. Scan file content for malicious patterns

VULN-002: CSV Injection (Formula Injection)
------------------------------------------------------------------------------
Severity: HIGH
Status: CONFIRMED
Location: app.py:440-443 (reports_export function)

Description:
The CSV export functionality directly concatenates user-controllable data
(sensor_id) into CSV output without sanitization.

Evidence:
    csv_lines.append(f'{r["timestamp"]},{r["sensor_id"]},{r["temperature_f"]},{r["humidity_percent"]}')

Risk:
If an attacker can inject a malicious sensor_id such as:
    =cmd|' /C calc'!A1
    @SUM(1+1)*cmd|' /C calc'!A0
    +cmd|' /C calc'!A0

When the CSV is opened in Excel, these formulas execute arbitrary commands
on the victim's machine (typically an administrator reviewing reports).

Mitigation Strategy:
1. Prefix cell values starting with =, @, +, -, tab, or carriage return with
   a single quote (') to prevent formula interpretation
2. Alternatively, use the csv module with proper quoting:

   import csv
   from io import StringIO

   def sanitize_csv_value(value):
       if str(value).startswith(('=', '@', '+', '-', '\t', '\r')):
           return "'" + str(value)
       return value

3. Add Content-Type header: text/csv; charset=utf-8
4. Consider adding Content-Disposition with warning filename

------------------------------------------------------------------------------
VULN-003: Missing IP-Based Rate Limiting
------------------------------------------------------------------------------
Severity: HIGH
Status: CONFIRMED
Location: app.py:126-184 (login function), utils/data_manager.py:110-128

Description:
Account lockout is implemented per-username only. There is no IP-based rate
limiting, allowing password spraying attacks where an attacker tries common
passwords against many usernames.

Evidence:
- Account lockout triggers after 4 failed attempts PER USERNAME
- No tracking of failed attempts by IP address
- No global rate limiting on login endpoint

Risk:
- Password spraying attacks bypass per-account lockout
- Credential stuffing attacks at scale
- Brute force attacks distributed across usernames

Mitigation Strategy:
1. Implement IP-based rate limiting using Flask-Limiter:

   from flask_limiter import Limiter
   limiter = Limiter(app, key_func=get_remote_address)

   @app.route('/login', methods=['POST'])
   @limiter.limit("5 per minute")
   def login():
       ...

2. Track failed login attempts by IP address
3. Implement progressive delays after repeated failures from same IP
4. Consider CAPTCHA after threshold of failed attempts
5. Log and alert on distributed attack patterns

VULN-004: Missing SESSION_COOKIE_SECURE Flag
------------------------------------------------------------------------------
Severity: HIGH
Status: NEW
Location: config.py (missing configuration)

Description:
The SESSION_COOKIE_SECURE flag is not set. This allows session cookies to be
transmitted over unencrypted HTTP connections.

Evidence (config.py):
    SESSION_COOKIE_HTTPONLY = True
    SESSION_COOKIE_SAMESITE = 'Lax'
    # SESSION_COOKIE_SECURE is NOT set

CLAUDE.md Requirement:
"Session cookies must include the Secure flag (transmitted over HTTPS only)"

Risk:
- Session hijacking via network sniffing on non-HTTPS connections
- Man-in-the-middle attacks can steal session tokens

Mitigation Strategy:
Add to config.py:
    SESSION_COOKIE_SECURE = True  # In production

For development, use environment-based setting:
    SESSION_COOKIE_SECURE = os.environ.get('FLASK_ENV') == 'production'

------------------------------------------------------------------------------
VULN-005: Missing HSTS Header
------------------------------------------------------------------------------
Severity: MEDIUM
Status: NEW
Location: app.py:49-62 (add_security_headers function)

Description:
HTTP Strict Transport Security header is not configured, allowing SSL
stripping attacks.

Evidence:
Security headers are set but HSTS is missing:
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['X-XSS-Protection'] = '1; mode=block'
    response.headers['Content-Security-Policy'] = ...
    # HSTS is NOT set

CLAUDE.md Requirement:
"HTTP Strict Transport Security (HSTS) headers should be enabled to prevent
SSL stripping attacks."

Mitigation Strategy:
Add to add_security_headers function:
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'

For production with HTTPS preloading:
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains; preload'

VULN-006: Password Hash Algorithm Non-Compliance
------------------------------------------------------------------------------
Severity: MEDIUM
Status: NEW
Location: config.py:32-33

Description:
The application uses PBKDF2-SHA256 for password hashing instead of bcrypt
as specified in CLAUDE.md requirements.

Evidence:
    PASSWORD_HASH_METHOD = 'pbkdf2:sha256'
    PASSWORD_HASH_ITERATIONS = 600000

CLAUDE.md Requirement:
"The bcrypt algorithm (via Werkzeug's generate_password_hash()) shall be
used for password hashing... A minimum work factor of 12 rounds shall be used"

Risk:
- PBKDF2 is acceptable but bcrypt is preferred for password hashing
- Non-compliance with documented security requirements
- bcrypt has better resistance to GPU-based attacks

Mitigation Strategy:
Change config.py:
    PASSWORD_HASH_METHOD = 'bcrypt'
    # Note: Werkzeug's bcrypt uses work factor 12 by default

Update data_manager.py create_user function:
    password_hash = generate_password_hash(
        password,
        method='bcrypt'  # Werkzeug handles salt and rounds
    )

Note: Existing user passwords will need migration or users will need to
reset passwords.

VULN-007: Username Enumeration via Error Messages
------------------------------------------------------------------------------
Severity: MEDIUM
Status: NEW
Location: app.py:150-181 (login function)

Description:
Different error messages reveal whether a username exists in the system,
enabling username enumeration attacks.

Evidence:
    if data_manager.is_account_locked(username):
        error = "Account is locked. Please try again later..."
    else:
        if data_manager.verify_password(username, password):
            # success
        else:
            attempts = data_manager.increment_failed_attempts(username)
            if attempts >= Config.MAX_LOGIN_ATTEMPTS:
                error = "Account has been locked due to too many failed attempts"
            else:
                error = f"Invalid username or password. {remaining} attempts remaining."

An attacker can distinguish:
- Locked accounts (username exists)
- Valid usernames with wrong passwords (shows remaining attempts)
- Non-existent usernames (shows generic error but no attempts increment)

Mitigation Strategy:
Use consistent error messages regardless of failure reason:
    error = "Invalid username or password"

For locked accounts, still show generic message externally but log the
actual reason. Consider implementing fake lockout behavior for non-existent
usernames to prevent timing-based enumeration.

VULN-008: Spoofable X-Forwarded-For Header
------------------------------------------------------------------------------
Severity: MEDIUM
Status: NEW
Location: utils/audit.py:22-27 (_get_client_ip function)

Description:
The X-Forwarded-For header is trusted directly without validation, allowing
attackers to spoof their IP address in audit logs.

Evidence:
    def _get_client_ip(self):
        if request.headers.get('X-Forwarded-For'):
            return request.headers.get('X-Forwarded-For').split(',')[0].strip()
        return request.remote_addr or 'unknown'

Risk:
- Attackers can hide their real IP in audit logs
- Circumvention of IP-based security controls (if implemented)
- Audit log tampering

Mitigation Strategy:
1. Only trust X-Forwarded-For when behind a known proxy
2. Configure trusted proxy IPs in config:

   TRUSTED_PROXIES = ['127.0.0.1', '10.0.0.0/8']

3. Use ProxyFix middleware with proper configuration:

   from werkzeug.middleware.proxy_fix import ProxyFix
   app.wsgi_app = ProxyFix(app.wsgi_app, x_for=1, x_proto=1)

4. Log both X-Forwarded-For and remote_addr for forensics

VULN-009: Missing Input Validation on Sensor ID
------------------------------------------------------------------------------
Severity: MEDIUM
Status: NEW
Location: app.py:476, data_manager.py:195-212

Description:
The sensor_id field accepts arbitrary user input with only whitespace
stripping. This enables CSV injection and potential log injection.

Evidence:
    sensor_id = request.form.get('sensor_id', 'SENSOR-001').strip()
    # No validation of content

Risk:
- CSV injection via malicious sensor_id values
- Log injection attacks
- Database pollution with invalid data

Mitigation Strategy:
1. Implement strict validation for sensor_id:

   import re
   SENSOR_ID_PATTERN = re.compile(r'^SENSOR-\d{3}$')

   def validate_sensor_id(sensor_id):
       if not SENSOR_ID_PATTERN.match(sensor_id):
           raise ValueError("Invalid sensor ID format")
       return sensor_id

2. Use a whitelist/dropdown of valid sensor IDs (already in ics.html template)
3. Reject or escape special characters in user-supplied sensor IDs

VULN-010: Insufficient Session Fixation Protection
------------------------------------------------------------------------------
Severity: LOW
Status: NEW
Location: utils/auth.py:79-97 (login_user function)

Description:
While session.clear() is called before setting new session data, Flask's
default client-side sessions don't guarantee a new session ID is issued.

Evidence:
    def login_user(username: str, role: str, last_login: str = None):
        session.clear()
        session['username'] = username
        session['role'] = role
        # ...

Risk:
If an attacker can set a known session cookie before the victim logs in,
the attacker may be able to hijack the session after login.

Mitigation Strategy:
1. Use server-side sessions (Flask-Session with Redis/database backend)
2. Explicitly regenerate session ID after authentication:

   from flask import session

   def login_user(...):
       session.clear()
       session.regenerate()  # If using Flask-Session
       # Or manually create new session identifier

3. Consider adding session binding to IP/User-Agent

VULN-011: Missing Password Complexity Requirements
------------------------------------------------------------------------------
Severity: LOW
Status: NEW
Location: utils/data_manager.py:59-86 (create_user function)

Description:
No password complexity requirements are enforced when creating users.

Evidence:
    def create_user(self, username: str, password: str, role: str) -> bool:
        # No password validation
        password_hash = generate_password_hash(password, ...)

Risk:
- Users may choose weak passwords
- Increased susceptibility to brute force attacks
- Does not meet typical compliance requirements

Mitigation Strategy:
Implement password validation:

    import re

    def validate_password(password):
        if len(password) < 12:
            raise ValueError("Password must be at least 12 characters")
        if not re.search(r'[A-Z]', password):
            raise ValueError("Password must contain uppercase letter")
        if not re.search(r'[a-z]', password):
            raise ValueError("Password must contain lowercase letter")
        if not re.search(r'\d', password):
            raise ValueError("Password must contain a digit")
        if not re.search(r'[!@#$%^&*(),.?":{}|<>]', password):
            raise ValueError("Password must contain special character")
        return True
